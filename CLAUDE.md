# Parquet Tools — Claude Code Context

## What This Project Is

A suite of CLI and browser-based tools for analysing industrial/embedded
system logs stored as Parquet files. Signals are time-series columns sampled
from CAN bus frames and Modbus registers. All timestamps are UTC.

Tools are invoked via `mise <taskname>` and implemented as standalone Python
scripts under `src/`. A shared library layer (`src/schema.py`, `src/loader.py`,
`src/derivatives.py`) is used by all tools — never duplicate logic across tools.

## Parquet Schema Conventions

> ⚠️  These are STUBS. Run `mise schema-interactive` with a .dbc or Modbus
> file to populate real signal names and update this section.

### Timestamp Column
- Column name: TBD — detect heuristically (see `src/schema.py:detect_timestamp_col`)
- Dtype: datetime64[ns, UTC] or float64 Unix seconds
- All tools normalise to float64 Unix seconds internally

### Signal Taxonomy (CAN / Modbus)
Signals fall into one of five categories. `schema.py` classifies each column:

| Category    | Description                              | Naming hints (heuristic) |
|-------------|------------------------------------------|--------------------------|
| `alert`     | Boolean 0/1 — fault/alert active        | parent frame OR signal name contains `fault` or `alert` |
| `measurement` | Numeric sensor readback               | everything else numeric  |
| `config`    | Fixed FW threshold, rarely changes      | ends in `_cfg` or `_config` |
| `command`   | User/operator command                   | ends in `_cmd` or `_command` |
| `control`   | Internal FW control signal              | ends in `_ctrl` or `_control` |

> **FW naming convention feedback template** (fill in and send to FW team):
> See `docs/fw_naming_conventions.md` — generated by `mise schema-interactive`.

### Alert Columns (boolean 0/1)
- `1` = fault/alert ACTIVE (SET) → display in **red**
- `0` = fault/alert INACTIVE (CLEAR) → display in **green**
- Detection: column classified as `alert` category AND dtype is bool/int with only {0,1} values
- Parent CAN frame name OR signal name must contain `fault` or `alert` (case-insensitive)
- Example hierarchy from real data:
  ```
  cabinet/data_can_signals_ALERTS/dcacCascadedControls_ALERTS/cell_mia
  cell6/circuitController_ALERTS_B_alerts/mv_aux_uvlo
  cell6/measurement_ALERTS_A_alerts/overtemp_dcdc_bot
  ```
  The hierarchy is encoded in column name prefixes or a separate metadata column — see schema.py.

### Numeric Signal Columns
- dtype: float32, float64, or integer
- Used by `whatchanged` and `whenchanged` tools
- `dsdt` = d(signal)/dt — derivative of any signal with respect to time (not voltage-specific)

## CLI Conventions

### Timestamp Format
- **Default**: Unix timestamp in seconds with decimal precision (e.g. `1710403267.123`)
- **ISO 8601 flag**: `--iso` on any command switches all timestamp output to ISO 8601 UTC
  (e.g. `2024-03-14T08:01:07.123Z`)
- Input timestamps (arguments) accept EITHER format automatically
- Conversion utility: `src/timeutils.py`

### Universal Flags (all tools must support these)
```
--file PATH       Path to .parquet file [env: PARQUET_FILE]
--iso             Output timestamps as ISO 8601 UTC instead of Unix seconds
--threshold N     Derivative spike multiplier for change detection [default: 2.0]
--help            Show usage, all options, and examples
```

### Exit Codes
- `0` = success
- `1` = bad arguments / missing file
- `2` = data error (schema mismatch, empty result, etc.)

### Colour Conventions (rich)
- `bold red`    = alert SET (0→1 transition)
- `bold green`  = alert CLEAR (1→0 transition)
- `yellow`      = warning / unknown state
- `cyan`        = signal name
- `dim`         = timestamp
- `orange1`     = sharp derivative spike (whatchanged)

## Derivative / Change Detection Algorithm

For `whatchanged`:
1. Load signal columns (numeric, non-alert) from the parquet file
2. For each signal, compute `dsdt[i] = (signal[i+1] - signal[i]) / (t[i+1] - t[i])`
   - Use `np.gradient` for uniform sampling (faster)
   - Use element-wise division for irregular timestamps
3. `baseline[signal] = mean(|dsdt|)` over the full file (or time window if specified)
4. A **sharp change** = `|dsdt[i]| > threshold * baseline[signal]`  (default threshold=2.0)
5. Report: signal name, timestamp of change, value_before, value_after, |dsdt| magnitude
6. `--before TIMESTAMP`: traverse backwards from timestamp, reverse-chronological output
7. `--after TIMESTAMP`: traverse forwards from timestamp, chronological output

For `whenchanged`:
- Rising edge: find first row where `signal >= aftervalue` preceded by `signal <= beforevalue`
- Falling edge: find first row where `signal <= aftervalue` preceded by `signal >= beforevalue`
- If `beforevalue < aftervalue` → rising edge search
- If `beforevalue > aftervalue` → falling edge search
- Debounce: ignore transitions that reverse within N samples (configurable, default N=3)

1. Run `mise run test` after any code change before declaring done
2. Never hardcode column names — always call `schema.py` detection functions

6. Keep analysis logic (derivatives.py, schema.py) completely separate from display logic (display.py)
